Qiskit v2.0 移行ガイド

Qiskit v2.0 は、Qiskit SDK の新たに安定化されたライフサイクルにおける次のステップです。 このメジャーバージョンは以前のバージョンの Qiskit との互換性を破りますが、 v1.0 リリースとは対照的に、パッケージ構造に変更はありません。

このガイドでは、Qiskit v2.0 における最も重要な機能変更の移行パスをモジュール別に説明します。 右側の目次から、関心のあるモジュールに移動してください。

Qiskit v2.0 のメリットについて詳しくは、IBM® ブログをご覧ください。

qiskit.circuit​

レガシー c_if 条件の削除​

Qiskit におけるクラシカル条件は、Qiskit v1.0 以前から IfElseOp 命令によって優先的に表現されてきました。 これらの操作は通常、コンテキストマネージャとして if_test メソッドを使用することで QuantumCircuit に追加されます。

これはまた、Instruction に condition 属性がなくなり、QuantumCircuit のゲート生成メソッド呼び出しの出力に c_if 呼び出しをチェーンできなくなったことを意味します。

現在、条件を持つ可能性がある命令は IfElseOp と WhileLoopOp のみです。 条件をテストする最も正確な方法は、isinstance チェックを使用して正しいクラスを持っているか検証することです。 ライブラリを Qiskit v2.0 以降に移行する間、レガシーな Instruction.condition 属性をサポートしたい場合は、getattr(operation, "condition", None) を使用できます。これにより、Instruction インスタンスは Qiskit v1.x 以前と同様に動作します。

Qiskit でシンプルなレジスタ等値条件をサポートするオリジナルのレガシーな方法は、個々の命令や命令セットに対して c_if メソッドを使用することでした。 これは完全な制御フロー形式よりも機能が厳密に劣っており、ハードウェア上でも十分にサポートされていませんでした。 このレガシーな方法は Qiskit v2.0 で削除されました。適切な条件を伴う if_test メソッドを使用してください。 レジスタの等値を表すために (register, integer) の 2 タプル形式を引き続き使用することも、より複雑な条件を表現するためにより表現力豊かなクラシカル計算サブシステムを使用することもできます。

たとえば、次のように記述していた場合：

from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)

最も直接的な変換は、if_test コンテキストマネージャで 2 タプル形式を使用することです：

from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)

この形式では、条件を再評価することなく、複数の操作に条件付きゲートがアクセスできます（インデントされたブロックにさらに追加するだけです）。 また、else 文を設けたり、より複雑な条件ロジックを使用したりすることもできます：

from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)

Instruction.duration および Instruction.unit の削除​

回路の Instruction オブジェクトには、以前は duration と unit という属性がありました。 これらは Qiskit v2.0 では存在しなくなりました。 これは、このフィールドが一般的にハードウェアの固定プロパティである情報を複製するだけであり、Target に明示的に格納されていたためです。 各命令に個別にこの情報を格納すると、Qiskit のメモリ要求が大幅に増加し、その情報に直接触れる必要のない回路操作でさえも遅くなっていました。

Delay 命令にはこれらのフィールドが引き続き含まれており、v2.0 で新たに追加された BoxOp も同様です。これらの命令は_可変_の遅延を持つためです。

ハードウェアでサポートされている特定の Instruction のハードウェア指定の持続時間を取得するには、Target にクエリを発行します。 これを行うには、まず操作名で Target にインデックスを付け、次にタプルとしてハードウェアキュービットのインデックスを指定します。 これにより、None（命令がサポートされているが特定のプロパティが割り当てられていない場合）または InstructionProperties インスタンスが返されます。後者の duration フィールドから、秒単位の持続時間（または指定されていない場合は None）を取得できます。

target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)

一般的に、クラシカル制御フローが回路実行の中でより一般的になるにつれ、回路の「持続時間」はますます明確でなくなります。 Qiskit v2.x には引き続き QuantumCircuit.duration 属性が含まれていますが、将来的に削除される可能性が高いため、使用を控えることを推奨します。この属性が有効であるための前提条件はますます頻繁に違反されるようになっています。 代わりに、新しい QuantumCircuit.estimate_duration メソッドを使用できます。これはより安全で、必要な前提条件についても明確です。

qiskit.transpiler​

generate_preset_pass_manager インターフェースの変更​

Qiskit v2.0.0 では、プリセットパスマネージャインターフェース（つまり、 generate_preset_pass_manager 関数と transpile 関数）に対する一連のアップデートが含まれており、 coupling_map や basis_gates などのルーズな制約入力の処理に影響します。 ルーズな制約が受け入れられるトランスパイルのユースケースの数が削減され、target および backend 入力（backend は Target インスタンスを含む BackendV2）が優先されます。これらの入力はこれらの制約をより表現力豊かに伝達できます。

以下の例は、主な変更点と移行のための推奨される代替案を示しています。以下の移行のヒントはすべて transpile と generate_preset_pass_manager の両方に適用されることに注意してください：

  1. backend_properties 入力引数はサポートされなくなり、BackendProperties クラスは削除されました。ゲートの持続時間とエラーを伝達するには、代わりに target または backend 引数を使用してください。カスタム Target を構築するための複数のパスがあります：

a. ターゲットをゼロから構築し、対応するゲートに関連するエラーと持続時間を追加します：

from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
from qiskit.circuit.library import XGate, CXGate

target = Target(num_qubits=2)
target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
target.add_instruction(
    XGate(),
    {
        (0,): InstructionProperties(error=.00001, duration=5e-8),
        (1,): InstructionProperties(error=.00002, duration=6e-8)
    }
)

pm = generate_preset_pass_manager(target=target)

b. GenericBackendV2 を使用して、事前に入力されたエラーと持続時間を持つ汎用ターゲットを素早く構築します。これらは構築後に更新することもできます：

from qiskit.providers.fake_provider import GenericBackendV2
from qiskit.transpiler import generate_preset_pass_manager

target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

pm = generate_preset_pass_manager(target=target)

  1. instruction_durations および timing_constraints 入力引数はサポートされなくなりました。代わりに target または backend 引数を使用してください。代替手段としては、カスタム Target をゼロから構築すること、汎用バックエンドを構築すること、そしてこれらに加えて Target.from_configuration メソッドを使用して instruction_durations および/または timing_constraints を含むカスタムターゲットを構築することが挙げられます：

    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)

  2. coupling_map および/または basis_gates を backend と共に指定することは推奨されず、v2.0 からは UserWarning が発生します。これらの場合、複数の情報源が存在し、バックエンドからのゲートエラーや持続時間などの情報を与えられた情報だけでは解決できません。 推奨される代替手段は、Target.from_configuration または GenericBackendV2 を使用して、ゼロから選択した制約を組み合わせたカスタムターゲットを定義することです。

  3. basis_gates 引数はカスタム基底ゲートを受け付けなくなりました。推奨される代替手段は target 引数で、ゼロから構築するか Target.from_configuration を使用して構築できます：

    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)

  4. カスタム coupling_map を 3 つ以上のキュービットを持つゲートを含む backend または basis_gates と組み合わせて指定する入力の組み合わせはサポートされなくなりました。カップリングマップはゲートの動作を決定するための十分な接続性の詳細を提供しません。このような場合、推奨される移行パスはゼロからカスタムターゲットを作成することです。

  5. inst_map 引数は代替手段なしに削除されました。これはトランスパイラにパルス情報を提供するために使用されており、qiskit.pulse モジュール全体が削除されました。

qiskit.pulse​

qiskit.pulse モジュールは、代替手段なしに Qiskit から完全に削除されました。 これには以下の関連項目が含まれます：

  • QuantumCircuit、DAGCircuit、DAGDependency のプロパティ calibrations。
  • QuantumCircuit のメソッド has_calibration_for および add_calibration。
  • DAGCircuit のメソッド has_calibration_for。
  • トランスパイラパス PulseGates、RXCalibrationBuilder、RZXCalibrationBuilder、EchoRZXWeylDecomposition、ValidatePulseGates。
  • qiskit.scheduler モジュールとそのすべての内容。
  • BackendV2 のメソッド instruction_schedule_map、drive_channel、measure_channel、acquire_channel、control_channel。
  • Target のメソッド has_calibration、get_calibration、instruction_schedule_map、update_from_instruction_schedule_map。
  • generate_preset_pass_manager、transpile、Target.from_configuration の inst_map 引数。
  • InstructionProperties の calibration フィールド。

これは、IBM Quantum® QPU がパルスレベルアクセスの提供を停止し、Qiskit チームが使用頻度の低いパッケージを維持するためのリソースを持たなくなったために起こりました。 qiskit.pulse モジュールは Qiskit の v1.x シリーズを使用している間は引き続き使用できます。v1.x シリーズは 2024 年 9 月までバグ修正サポート、2025 年 3 月までセキュリティサポートがありますが、v2.x シリーズからはモジュールが削除されています。

qiskit.pulse のほとんどの用途は、IBM Quantum Heron QPU（および該当する場合は後続世代）のフラクショナルゲートへの直接アクセスで置き換えることができます。 詳しくはパルス使用のフラクショナルゲートへの移行ガイドをご覧ください。

Qiskit Pulse の削除により、Qiskit v2.0 では ScheduleBlock オブジェクトを含む QPY ファイルを読み込むことができなくなりました。 これらのファイルは Qiskit v1.x シリーズで引き続き読み込むことができます。

Qiskit 2によるDiVincenzo基準ハンズオン入門

はじめに​

物理学者David DiVincenzoは、量子コンピュータのあらゆる物理的実装に必要な5つの主要要件と、量子通信のための2つの追加基準を示しました。このノートブックでは、実践的なQiskitデモンストレーションを通じて、各DiVincenzo基準を体験します。理論の深掘りは行わず、各セクションで1つの基準を簡潔に説明したうえで、Qiskit 2を使ったコード演習を提供します。シミュレータや実際のIBM Quantumデバイスでサーキットを実行し、各原理をハンズオンで探求していただきます。

DiVincenzoの量子計算に関する5つの基準：

  1. よく特性評価されたQubitを持つ、スケーラブルな物理システム。
  2. Qubitを単純な基準状態（例：|00…0〉）に初期化する能力。
  3. 長いデコヒーレンス時間（Qubitのコヒーレンスがゲート操作時間よりもはるかに長いこと）。
  4. 普遍的な量子Gateのセット（任意のユニタリ演算を実行できること）。
  5. Qubit固有の測定能力（各Qubitの状態を読み出すこと）。

(DiVincenzoは量子通信のための2つの基準も述べています。定常Qubitと「フライング」Qubitを相互変換する能力、およびフライングQubitを場所間で忠実に転送する能力です。これらはこのノートブックの最後の推奨アクティビティに含まれています。)

以下の各セクションは1つの基準に対応しています。Qiskitを使い、コードとインタラクティブな実験でそのコンセプトを説明します。たとえば、Qubitの数やCircuitの深さを増やすとどのように結果が変わるか（基準1）、Qubitの状態のリセットと準備の方法（基準2）、シミュレータと実デバイスでのQubitsの測定方法（基準4）、Qiskitが普遍Gateをどのように構成するか（基準3）、有限のコヒーレンス（T₁、T₂）がどのように計算に影響するか（基準5）を見ていきます。最終的には、各DiVincenzo基準が実際に何を意味し、Qiskitがどのようにそれらの実験を可能にするかについて、より深い直感を得ることができるでしょう。

1. 基準1 – スケーラブルでよく特性評価されたQubit​

基準1： 「よく特性評価されたQubitを持つ、スケーラブルな物理システム。」 これは、Qubit数を増やしても信頼性高く制御できる量子ハードウェアプラットフォームが必要であることを意味します。各Qubitの特性（エネルギー準位、エラー率、接続性など）がよく理解されている必要があります。本質的には、システムが崩壊することなく、より大きなCircuitを構築したいのです。実際には、Qubit数やCircuitの深さをスケールアップすると、エラーとデコヒーレンスが蓄積されるため、スケーラビリティを実証することは、サイズの増加がパフォーマンスにどう影響するかを理解することも意味します。

デモの目標： Qiskitを使って、Circuitのスケールアップ（Qubit数またはGateの深さ）が出力の忠実度に与える影響を示します。理想的なシナリオとノイズのあるシナリオをシミュレートし、システムや深いCircuitが大きくなるにつれてデコヒーレンスやエラーにどう影響されるかを見てみましょう。

まず、3つのQubitで小さなエンタングル状態（GHZ状態）を構築し、次に5つのQubitでより大きな状態を構築して、シンプルなスケーリングテストとします。n個のQubitのGHZ状態は 12(∣0...0⟩+∣1...1⟩)\frac{1}{\sqrt{2}}(|0...0\rangle + |1...1\rangle)2​1​(∣0...0⟩+∣1...1⟩) です。理想的なシミュレーションでは、n-QubitのGHZを測定すると、等確率の2つの結果（すべて0またはすべて1）のみが得られます。nまたはCircuitの深さを増やすにつれて、理想的な出力とノイズのある出力を比較します。

from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator
from qiskit.visualization import plot_histogram
from qiskit.transpiler.preset_passmanagers import generate_preset_pass_manager
from qiskit_ibm_runtime import SamplerV2 as Sampler

# 3-qubit GHZ circuit
qc3 = QuantumCircuit(3, 3)
qc3.h(0)
qc3.cx(0, 1)
qc3.cx(1, 2)
qc3.measure([0, 1, 2], [0, 1, 2])

# 5-qubit GHZ circuit (scaling up the number of qubits)
qc5 = QuantumCircuit(5, 5)
qc5.h(0)
qc5.cx(0, range(1, 5))    # entangle qubit 0 with all others
qc5.measure(range(5), range(5))

# Transpile for a simulator backend
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc3 = pm.run(qc3)
isa_qc5 = pm.run(qc5)

# Run ideal simulations (no noise)
sampler = Sampler(mode=sim_backend)

job3 = sampler.run([isa_qc3], shots=1024)
result3 = job3.result()
counts3 = result3[0].data.c.get_counts()

job5 = sampler.run([isa_qc5], shots=1024)
result5 = job5.result()
counts5 = result5[0].data.c.get_counts()

print("3-qubit GHZ counts (ideal):", counts3)
plot_histogram(counts3, legend=['3-qubit ideal'], figsize=(6,4))
print("5-qubit GHZ counts (ideal):", counts5)
plot_histogram(counts5, legend=['5-qubit ideal'], figsize=(6,4))

期待される結果（理想的なケース）： 3-Qubit GHZは理想的には、カウント内でおよそ50%の000と50%の111を生じさせます。5-Qubit GHZは約50%の00000と50%の11111を生じさせます。状態が理想的に完全にコヒーレントでエンタングルされているため、他のビット列は現れません。各Circuitのヒストグラムに、すべてゼロとすべてイチの結果に対応する2本の高いバーが表示されるはずです。

次に、ノイズのある環境ではどうなるかを見てみましょう。Qiskit Aerのノイズモデル機能を使って、実際のデバイスのエラーを模倣します。たとえば、IBMバックエンドの特性を使って、Gateエラー、有限のGate時間、Qubitの緩和（T₁）、位相緩和（T₂）、および読み出しエラーを含むノイズモデルを作成できます。ここでは、IBM Quantum Brisbaneデバイスを表すフェイクバックエンドを使用してノイズモデルを生成し、そこにGHZ Circuitを再実行します。

2. 基準2 – Qubitの初期化​

基準2： 「|000…〉のような単純な基準状態にQubitの状態を初期化する能力。」 すべてのQubitは既知の参照状態（通常は各Qubitのグラウンド状態|0〉）から確実に開始される必要があります。初期化は、アルゴリズムがクリーンな状態から開始できるようにするために不可欠です。実際には、IBM量子デバイスでは、各Circuit実行の開始時に各Qubitが自動的に|0〉にリセットされます。Qiskitはまた、計算中にQubitをリセットしたりカスタム状態を準備したりする命令も提供しています。

デモの目標： Qiskitでの開始時および回路途中でのQubitの初期化方法を示します。reset命令と状態準備メソッドの使用方法をデモします。

演習2：特定の状態の準備​

以下のコードブロックで、∣10⟩|10\rangle∣10⟩の状態を準備するためのQuantumCircuitを完成させてください。これは、Qubit 0が状態∣0⟩|0\rangle∣0⟩、Qubit 1が状態∣1⟩|1\rangle∣1⟩であることを意味します。これを達成するために適切なGateと命令を使用してください。

from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator

# Create a circuit to initialize qubits to |10> and verify by measurement
qc_init = QuantumCircuit(2, 2)

# --- YOUR CODE HERE ---

# 1. Set qubit 1 to the |1> state
qc_init.x(1)

# 2. Explicitly reset qubit 0 to the |0> state
qc_init.reset(0)

# --- END YOUR CODE ---

qc_init.measure([0, 1], [0, 1])
qc_init.draw('mpl')
# Run the circuit and check the outcome
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_init = pm.run(qc_init)

sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_init], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Outcome of |10> state measured in Z-basis:", counts)
plot_histogram(counts)

シミュレーションから、10（Qubit1=1、Qubit0=0のバイナリ）が100%の確率で表示されるはずです。これはQubit 1が|1〉に、Qubit 0が|0〉に正常に準備されたことを意味します。

より一般的な状態準備については、Qiskitのinitializeメソッドを使って任意の状態への初期化が可能です。たとえば、重ね合わせ状態である∣+⟩=(∣0⟩+∣1⟩)/2|+\rangle = (|0\rangle+|1\rangle)/\sqrt{2}∣+⟩=(∣0⟩+∣1⟩)/2​の状態にQubitを準備し、2つのQubitをベル状態(∣00⟩+∣11⟩)/2(|00\rangle+|11\rangle)/\sqrt{2}(∣00⟩+∣11⟩)/2​に準備してみましょう：

import numpy as np

# Initialize a single qubit in |+> state and measure in Z-basis
qc_plus = QuantumCircuit(1, 1)
state_plus = [1/np.sqrt(2), 1/np.sqrt(2)]   # amplitude for |0> and |1>
qc_plus.initialize(state_plus, 0)
qc_plus.measure(0, 0)

# Initialize two qubits in a Bell state manually
qc_bell = QuantumCircuit(2, 2)
bell_state = [1/np.sqrt(2), 0, 0, 1/np.sqrt(2)]  # amplitudes for |00>,|01>,|10>,|11>
qc_bell.initialize(bell_state, [0, 1])
qc_bell.measure([0, 1], [0, 1])

# Transpile and run the initialization circuits
isa_qc_plus = pm.run(qc_plus)
job_plus = sampler.run([isa_qc_plus], shots=1024)
result_plus = job_plus.result()
counts_plus = result_plus[0].data.c.get_counts()

print("Outcome of |+> state measured in Z-basis:", counts_plus)

isa_qc_bell = pm.run(qc_bell)
job_bell = sampler.run([isa_qc_bell], shots=1024)
result_bell = job_bell.result()
counts_bell = result_bell[0].data.c.get_counts()

print("Outcome of Bell state measured in Z-basis:", counts_bell)

期待される結果： 単一Qubitの|+〉状態は、測定するとそれぞれ約50%の確率で0と1を生じさせます。ベル状態の測定は約50%の00と50%の11を生じさせるはずです。これらが確認できれば、それらの状態への初期化が成功したことが確認されます。

回路途中の初期化： Qiskitのresetは、Circuit途中で使用して、Qubitをオンザフライで|0〉に再初期化できます。たとえば、エラー訂正コードや反復アルゴリズムでは、Qubitを測定してから再利用のためにリセットすることがよくあります。reset操作は決定論的で、既存の状態を廃棄してQubitをグラウンド状態に冷却します。

デバイスの例： ibmq_brisbane（127 Qubit）などのハードウェアやIBMデバイスでは、ジョブが実行されるときにすべてのQubitがデフォルトで|0〉から始まります。別の開始状態が必要な場合は、最初にGateを適用します（|1〉を得るためにXを使用したように）。継続的な再初期化（量子エラー訂正のための）は、素早く行うことが困難であるため、現在も活発な研究トピックです。幸い、基本的な使用では|0…0〉からクリーンに開始する能力が利用可能であり、他の望ましい開始状態を達成する方法も示しました。

3. 基準3 – 長いコヒーレンス時間（デコヒーレンスとゲート時間）​

基準3： 「ゲート操作時間よりもはるかに長い、適切なデコヒーレンス時間。」 これは、必要な操作を実行するのに十分な時間、Qubitが量子状態を維持する必要性に対処するものです。各Qubitには T₁時間（エネルギー緩和時間：|1〉が|0〉に崩壊する速さ）とT₂時間（デファイジング時間：相対的な位相コヒーレンスが失われる速さ）があります。量子コンピュータが正しく動作するためには、これらの時間スケールがゲート操作の持続時間をはるかに上回る必要があります。

デモの目標： 実行時間が長くなるにつれてデコヒーレンスがCircuitの結果にどう影響するかを示すことで、Qiskit内のQubitコヒーレンスを調査します。既知のT1/T2時間を持つフェイクBackendを使用して、この効果をシミュレートします。

有限コヒーレンスの影響を実証するために、T1崩壊実験をシミュレートします。Qubitを|1〉状態に準備し、delay命令を使用してしばらく待機してから、測定します。遅延時間が増加するにつれて、|1〉を測定する確率が低下することが期待されます。

# This part is done for you. We are creating a list of circuits,
# each with a different delay time.

time_delays_ns = [0, 50000, 100000, 150000, 200000, 250000, 300000]  # delay durations in ns

decay_expts = []
for delay in time_delays_ns:
    qc = QuantumCircuit(1, 1)
    qc.x(0)  # initialize qubit to |1>
    if delay > 0:
        qc.delay(delay, 0, unit='ns')  # wait 'delay' nanoseconds
    qc.measure(0, 0)
    decay_expts.append(qc)

decay_expts[1].draw('mpl') # Visualize one of the circuits

4. 基準4 – 普遍的な量子Gateのセット​

基準4： 「量子Gateの『普遍的な』セット。」 これは、有限の基本Gateのセットを組み合わせることで、ハードウェアが任意の量子計算を実行できる必要があることを意味します。古典コンピュータではNANDが普遍的ですが、量子においては普遍的なGateセットの選択肢が多数あります（例：{H, T, CNOT}または特定のマシンのネイティブGate）。たとえば、IBMデバイスには特定のQubit間の任意の単一Qubit回転とCNOTなどのネイティブ操作のセットがあり、これらを組み合わせることで普遍的になります。Qiskitの役割はしばしば、高レベルのGateをこれらの基底Gateにコンパイルすることです。

デモの目標： GateがQiskitによってどのように分解されるかを示すことで、Gateの普遍性を説明します。ネイティブでないGate（3-Qubit Toffoli Gate、CCXなど）を取り上げ、それがデバイスの基底Gateにどのように分解されるかを見ます。これにより、提供されたGateセットが確かに普遍的であること、つまりより複雑な操作を生成できることが示されます。

まず、典型的なIBM BackendのベースGateを確認しましょう。デバイス（またはそのフェイクバージョン）の設定をクエリします。たとえば、ibmq_brisbaneの基底Gateは以下の通りです： P(qubit=1)の確率が遅延時間の増加とともに低下し、T1緩和を特徴とする指数的崩壊曲線をたどることが確認できるはずです。これは、Circuitの実行時間が長すぎる場合に有限コヒーレンス時間がいかに計算エラーを引き起こすかを直接示しています。

アルゴリズムへの影響： 長いアルゴリズム（多数の連続したGateを含む）を試みると、総実行時間がT2に近づくかそれを超え、終了前に状態がコヒーレンスを失う可能性があります。そのため、コヒーレンス時間の改善とGateの高速化が、量子ハードウェア研究における最も重要な目標の二つです。

from qiskit_ibm_runtime.fake_provider import FakeBrisbane
fake_brisbane = FakeBrisbane()
print("Basis gates for ibmq_brisbane:", fake_brisbane.configuration().basis_gates)

これにより、['id', 'rz', 'sx', 'x', 'ecr']のような出力が得られる場合があります。これらはハードウェアがネイティブにサポートするプリミティブ操作です（Identity/no-op、RZ回転、sqrt(X) Gate、X Gate、そしてcontrolled-X）。他のGateはこれらから構成する必要があります。このセットは量子計算において普遍的であることが知られています（本質的に単一Qubit回転とエンタングルする2-Qubit Gateが普遍的なセットを形成します）。

次に、テストケースとしてToffoli（CCX）Gateを取り上げます。CCXは、2つの制御Qubitが両方とも1である場合にのみターゲットQubitを反転させます。これはIBMハードウェアのネイティブGateではありません。Qiskitはccx命令を提供していますが、内部的にはそれを分解します。

演習4：Toffoli Gateの分解​

以下のコードを完成させて、Toffoli（CCX）Gateを含むCircuitを構築し、Qiskitを使用してFakeBrisbane BackendのネイティブベースGateに分解してください。

from qiskit import QuantumCircuit
from qiskit_ibm_runtime.fake_provider import FakeBrisbane

# The fake_brisbane backend from the previous cell is reused here.

# --- YOUR CODE HERE ---

# 1. Create a circuit that can accommodate a Toffoli gate
qc_toffoli = QuantumCircuit(3)

# Apply a CCX gate with controls on qubits 0, 1 and target on qubit 2
qc_toffoli.ccx(0, 1, 2)

# 2. Transpile the circuit to the fake Brisbane backend
pm = generate_preset_pass_manager(backend=FakeBrisbane(), optimization_level=1)

isa_qc_toffoli = pm.run(qc_toffoli)

# --- END YOUR CODE ---

print("Toffoli circuit before decomposition:")
print(qc_toffoli)

print("\nToffoli circuit after transpiling to Brisbane basis:")
# The .draw() method will now show the decomposed circuit
print(isa_qc_toffoli.draw(fold=120))

トランスパイル後の出力では、CCXがrz、sx、ecrなどのより基本的なGateのシーケンスに置き換えられているはずです。これにより、ネイティブGateがToffoliを表現するのに十分であることが証明されます。

実践における普遍性： 上記の演習は、複雑な3-Qubit Gateがよりシンプルなものから構築されたことを示しています。一般に、任意の多Qubitユニタリは1-Qubitおよび2-Qubit Gateから構成できます。Transpilerは量子ソフトウェアスタックの重要なコンポーネントであり、実行したい抽象的なアルゴリズムと特定の量子デバイスが実際に実行できる物理的な操作との間のギャップを埋めます。

デバイスの例： ibmq_brisbane デバイスはEagleアーキテクチャを使用し、上記に示した基底Gateを持ちます。つまり、それらのマシンに送られるアルゴリズムはすべて、それらの操作のシーケンスに変換されます。この基準は本質的に制御可能性に関するもので、Qubitに対して必要な操作を実行するための十分な制御手段を持っていることを意味します。

5. 基準5 – Qubitの測定​

基準5： 「Qubit固有の測定能力。」 各Qubitの状態は測定可能でなければなりません（通常は計算基底、|0〉または|1〉で）。言い換えると、量子Circuitを実行した後、各Qubitを0/1の古典的なビットとして読み出す必要があります。この基準は、各Qubitに対して信頼性の高い検出器を持ち、測定するQubitを選択できることに関するものです。

デモの目標： Qiskitでシミュレータおよび実際のデバイスで測定を実行する方法を示し、違い（測定ノイズなど）を強調します。さまざまな状態にあるいくつかのQubitを測定し、結果を検討します。また、シミュレータとハードウェアの結果を比較することで、読み出しエラーがどのように現れるかを示します。

まず、シンプルな測定の例を見てみましょう：

qc_measure = QuantumCircuit(2, 2)
qc_measure.x(0)              # qubit 0 -> |1>, qubit 1 stays |0>
qc_measure.measure([0, 1], [0, 1])
qc_measure.draw('mpl')
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)
job = sampler.run([isa_qc_measure], shots=1000)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulator measurement counts:", counts)

シミュレータでは01の1000カウントが期待されます。次に、これをシミュレートすることで測定エラーの影響を見てみましょう。AerシミュレータにReadoutErrorを追加することができます。Qiskit Aerでは、ReadoutErrorを定義してノイズモデルのQubitに付加することができます。

演習5：読み出しエラーのシミュレート​

各Qubitが2%の確率で誤って測定される（0が1として読み取られる、または1が0として読み取られる）シンプルな読み出しエラーモデルを定義するコードを完成させてください。次に、このノイズモデルで測定Circuitを実行してください。

from qiskit_aer.noise import NoiseModel, ReadoutError

# --- YOUR CODE HERE ---

# 1. Define a 2% readout error for each single qubit.
# [[P(0|0), P(1|0)], [P(0|1), P(1|1)]]
ro_error = ReadoutError([[0.98, 0.02], [0.02, 0.98]])

# 2. Create a new noise model
noise_model_ro = NoiseModel()

# 3. Add the readout error to all qubits in the noise model
noise_model_ro.add_all_qubit_readout_error(ro_error)

# --- END YOUR CODE ---

sim_backend.set_options(noise_model=noise_model_ro)
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)

# Run the measurement circuit with readout noise
sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_measure], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulation with 2% readout error:", counts)

このシミュレートされた出力は、実際のハードウェアが生成する可能性のあるものと同様に、誤ったカウント（11、00、10など）を示し、不完全な測定の影響を実証します。

デバイスの例： ibmq_brisbaneなどの実際のデバイスでは、同じCircuitを実行すると、不正確な結果に対して同様のゼロでないカウントが見られる可能性が高いです。デバイスのキャリブレーションデータには、各Qubitの読み出しエラーが記載されています。特定のQubitをターゲットにして読み取る能力は非常に重要であり、そのエラー特性を理解することが意味のある結果を得るための鍵です。実際のデバイスでの実行については、演習1b：実際のIBM Quantum コンピュータでの実行で示しました。

量子通信基準（フライング・Qubit）​

DiVincenzoは、ネットワーク接続された量子コンピュータを構築する場合に重要な、量子通信に特化した2つの基準も挙げています：

  1. 静止Qubitとフライング・Qubitを相互変換する能力。（例：プロセッサ内のQubitを移動できる光子にマッピングする。）
  2. フライング・Qubitを場所間で忠実に伝送する能力。（例：量子情報を失わずに光ファイバーを通じて光子Qubitを送信する。）

Qiskitは主にチップ上の静止Qubitを扱うため、これらは標準的なQiskitの使用範囲を超えています。ただし、量子テレポーテーションというシンプルな例でこれらの基準の概念を示すことができます。テレポーテーションは、静止Qubitの状態をエンタングルしたペア（「フライング」部分）によって運ばれる情報と古典的な通信に変換し、それを使用して別の場所にある別の静止Qubitで状態を再構成することを示しています。

推奨アクティビティ：Qiskit in Classrooms の量子テレポーテーションモジュールを受講する​

Katie McCormick博士によるQiskit in Classrooms 量子テレポーテーションモジュールは、量子情報において最も魅力的なプロトコルの一つである量子テレポーテーション（エンタングルメントとわずか2つの古典ビットを使用して、AliceからBobに量子状態（Qubit）を送信する）を順を追って案内します。エンタングルしたBellペアの準備方法、Aliceの側でBell基底測定を実行する方法、古典的な結果を送信する方法、そしてBobのQubitに正しい量子Gateを適用して元の状態を完全に回復する方法など、テレポーテーション手順全体をステップバイステップで学びます。その過程で、Qubitをテレポートさせる情報が不可複製定理に違反したり光速を超えたりしない理由も探求します。IBM Quantumハードウェアまたはシミュレータを使用したハンズオン演習を通じて、実際の測定、エンタングルメント、フィードフォワード制御の実用的な理解を深めます。

量子テレポーテーションをマスターすることで、異なるノード間で量子情報をエンコード、送信、回復する方法を理解し、量子ネットワーク、リピータシステム、安全な通信スキーム、およびスケーラブルなモジュール型量子コンピューティングの基盤を築くことができます。 これが基準6と7にどう関連するか： 実際の量子ネットワークでは、共有されたエンタングルペアは「フライング」Qubit（光子など）をAliceとBobの場所間に分散させることで生成されます（基準7：忠実な伝送）。テレポーテーションプロトコル自体は、AliceのQubit状態をエンタングルペアの彼女の半分にマッピングし、効果的にBobに「送信」する方法として機能します（基準6：相互変換）。Qiskitはプロトコルのロジックを完全にシミュレートでき、これらの基準が通信アーキテクチャでどのように満たされるかの概念モデルを提供します。

まとめ​

私たちは、Qiskitを使用してDiVincenzo基準を説明することに焦点を当てた一連のコード演習を設計しました。これらのハンズオンの例を通じて、実際の量子コンピュータプラットフォームが各要件をどのように満たすかを探求しました：

  • スケーラビリティ：より多くのQubitでCircuitを構築し、ノイズのスケーリングを理解する。
  • 初期化：リセットと状態準備を使用して、既知の状態で確実に計算を開始する。
  • 普遍的なGate：複雑な操作をマシンの基底Gateにトランスパイルし、任意の計算を実行できることを証明する。
  • 測定：Qubitを読み出し、現実的な読み出しエラーに対処する。
  • コヒーレンス：アルゴリズムの忠実度に対する有限T₁、T₂の影響と、デコヒーレンスに対して操作が速い必要性を確認する。

完全性のために、Qiskit in Classrooms 量子テレポーテーションモジュールを通じて量子通信の側面にも触れ、最後の2つの基準（フライング・Qubit）と結びつけました。

最後に、これらの基準がIBMの実際の量子コンピュータでどのように組み合わさるかを指摘する価値があります。ibmq_brisbaneのようなデバイスは127の超伝導Qubit（基準1）を持ち、各Qubitは|0〉から開始し（基準2）、普遍性のためのキャリブレートされたGateセットとコンパイラ（基準4）、各Qubitのマイクロ波読み出し共振器（基準5）、そしてナノ秒単位の操作に対して数百マイクロ秒オーダーのコヒーレンス時間（基準3）を持っています。量子ネットワーキング実験のために、IBMや他の機関はフライング・Qubitのためのマイクロ波から光学への変換や、遠隔Qubitのエンタングル（基準6と7）を探求しています；これらは活発な研究分野です。

このノートブックの演習を完了することで、DiVincenzo基準の定義を見るだけでなく、コードを通じて実際に体験したことになります；実際の量子ハードウェアとアルゴリズムに対して各要件が何を意味するかの直感を築いています。これらの実験を自由に拡張して、量子コンピューティングをお楽しみください！
